BirbGames Admin Settings Guide

This document provides a detailed breakdown of the Settings section of the Birb Admin Backend. This is the central configuration panel (or the "central nervous system") for the entire platform, where global settings, appearance, features, and all add-on parameters are managed.

The Settings interface is organized by the navigation panel on the left, which is split into two main components: Main Configuration categories and individual Add-on Configuration links.

I. Main Configuration Categories (Global Settings)

These categories control the core, site-wide functionality and visual identity of the entire platform.

GENERAL

The GENERAL section controls the fundamental public identity and metadata of the platform. Changes here affect how the platform is presented to users and external services.

  • Purpose: To define the core brand and assets for the application.
  • Key Parameters:
    • App name: The name displayed across the platform (e.g., BirbGames).
    • App URL: The main domain of the application (e.g., https://birbgames.com).
    • Logo: The main visual asset for the brand identity.
    • OG (Open Graph) Image: The image used when links to the site are shared on social media platforms like Twitter or Facebook.
    • Favicon: The small icon displayed in the browser tab.

THEME

The THEME section offers powerful controls for customizing the visual appearance, typography, and advanced code injection for the entire user interface.

  • Purpose: To manage the site's aesthetics and integrate custom client-side code.
  • Key Fields and Customization Options:
    • Template: Selects the foundational structural layout of the platform (e.g., Stakely).
    • Theme: Selects the aesthetic preset or color scheme applied to the chosen template (e.g., Nebula Mist).
    • Fonts (Typography Control): This allows for complete control over typography by injecting custom font sources.
      • Body font URL / Heading font URL: Links to external font services (e.g., Google Fonts API URL) for fetching custom fonts.
      • Body font family name / Heading font family name: The corresponding CSS font-family names required to apply the loaded fonts (e.g., Play, Goldman).
    • CSS (Custom CSS rules): A text area for injecting custom CSS styles directly into the site for granular design adjustments.
    • JavaScript (Custom scripts): A text area for injecting custom JavaScript code, primarily used for integrating third-party trackers (like analytics), widgets, or custom client-side logic.
  • Management: Both Custom CSS and Custom Scripts include SAVE and DELETE options for field management.

FEATURES

The FEATURES section acts as a master control panel for enabling, disabling, and fine-tuning the operational parameters of nearly every user-facing system on the platform. It is segmented into top-level notification controls and numerous expandable sub-sections.

  • Purpose: To globally enable, disable, and configure core platform functionalities, user experience elements, and background services.

Admin Notifications

This section defines when and how administrators are alerted to critical events on the platform.

  • Send to: The recipient address or group for notifications (requires SMTP server setup in the Services tab).
  • Notify when a new user signs up: Toggles immediate alerts for new account creation.
  • Notify when a new KYC request is submitted: Toggles alerts when a user submits verification documentation.
  • Notify when a user wins/loses a game: Toggles alerts for high-value win/loss events, configurable via:
    • Win threshold: The minimum win amount (e.g., 1000) that triggers a notification.
    • Loss threshold: The minimum loss amount (e.g., 1000) that triggers a notification.

Expandable Subsections (Detailed Feature Configuration)

The majority of the FEATURES section consists of the following expandable subsections, which contain numerous toggles and parameter fields for specific system behaviors:

Affiliate program

Purpose: Manages the configuration of the user referral and affiliate system, including commission rates and payment logic.

  • Key Parameters:
    • Enabled: A single toggle switch that acts as the master control to globally turn the entire Affiliate program on or off.
    • Commissions auto approval frequency: A dropdown to schedule when accrued affiliate commissions are automatically approved for payout (e.g., Every week on Monday at 00:00).
    • Do not validate IP address when creating commissions: A security toggle. If enabled, the system will ignore the IP address when calculating commissions, which may be necessary for some affiliate arrangements but increases the risk of self-referral fraud.
    • Commissions (Tiers): This section defines the specific commission structure based on multi-tiered membership (Tier 1, Tier 2, Tier 3, etc.). These rates are applied to the referring affiliate based on the referred user's activity.
      • Tier 1 (Base Level Commissions): Defines the rates for a direct referral.
        • Sign up: Fixed amount awarded upon user registration (e.g., 100 CREDIT).
        • Email verification: Fixed amount awarded upon user verifying their email (e.g., 50 CREDIT).
        • Game loss: Percentage commission rate applied based on the referred user's net loss (e.g., 10 % of bet amount).
        • Game win: Percentage commission rate applied based on the referred user's net win (e.g., 10 % of bet amount).
        • Raffle ticket: Percentage commission rate applied to the cost of raffle tickets purchased by the user (e.g., 20 % of raffle ticket cost).
        • Deposit: Percentage commission rate applied to the total amount deposited by the user (e.g., 5 % of deposit amount).
      • Tier 2 / Tier 3: These expandable sections allow the administrator to define separate, typically lower, commission rates for indirect referrals (referrals made by the affiliate's referrals), enabling a multi-level marketing (MLM) structure.

Authentication

Purpose: Controls the registration, login, and security policy requirements for all user accounts, ensuring system integrity.

  • Key Parameters:
    • Enable email authentication: Master switch to allow or disallow users to register and log in using an email address and password. If disabled, users must rely solely on configured social login methods.
    • Require email verification: Forces new users to click a link sent to their email address before their account becomes fully active, reducing spam accounts and validating user identity.
    • Auto-generate username on registration: If enabled, the system automatically assigns a username to new registrants, streamlining the sign-up process.
    • Validate device fingerprint: A critical security feature. If enabled, the system tracks a unique ID for the device used during registration/login. A user will be prevented from logging in or registering with a device that is already linked to a different existing account, which is highly effective against bot farms and multi-accounting fraud.

Bet stream

Purpose: Configures the live, real-time betting feed visible to users, controlling which bets are displayed and how the feed operates.

  • Key Parameters:
    • Display count: Defines the maximum number of bet records (e.g., 10) that are visible in the live stream at any given time.
    • Big bet threshold: Sets the minimum wager amount (e.g., 100) that qualifies a bet as a "Big Bet," which may receive special visual treatment or prioritization in the feed.

Bots

Purpose: Manages the configuration and behavior of any simulated or promotional player accounts. This section is divided into two primary areas: Persistent and Temporary bets.

    • Persistent bets: Used for ongoing, scheduled simulation of user activity to maintain a lively appearance in game lobbies.
      • Frequency: Defines how often the bots will "wake up" to place a bet (e.g., Every hour).
      • Min bets / Max bets: Sets the minimum and maximum number of bots that will be active in a single cycle.
      • Min bet / Max bet: Allows setting custom minimum and maximum bet limits for bots, overriding the limits set in the individual game configurations if left empty.
    • Temporary bets: Used for setting up the characteristics of automated bets based on randomized parameters.
      • Bet amounts: A checklist or selection field allowing the administrator to define a list of valid bet sizes the bots can use (e.g., 1, 2, 3... 500).
      • Win probability: A slider or field to set the target probability of a bot win, influencing the visual flow of the simulated bet stream.
      • Interval: Sets the minimum and maximum time interval (in milliseconds, e.g., 500ms to 1000ms) between sequential bot bets. This controls the speed of the activity feed.
      • Max multiplier: Defines the highest possible win multiplier a bot can achieve, which prevents unrealistic simulation of jackpot wins.

Chat

Purpose: Controls the overall availability and functional requirements of the user-to-user chat system.

  • Key Parameter:
    • Enabled: A single toggle switch that acts as the master control to globally turn the chat system on or off. The accompanying note ("Set up broadcaster to use this feature.") indicates that an external service for real-time communication (like Pusher or a similar broadcaster configured in the SERVICES tab) is required for this feature to function, even when enabled.

Games

Purpose: Manages general asset and card deck configurations that are applicable across all original card games on the platform.

  • Key Parameters:
    • Card deck: A dropdown menu to select the graphical style and type of card deck assets to be used exclusively by original card games (such as Blackjack, Caribbean Poker, and Casino Hold'em). The available options are:
      • Classic: Standard, traditional look.
      • French: A style often featuring different indices or suits than the classic English/American deck.
      • Poker: The default, high-contrast deck optimized for poker variants.
      • Slick: A modern, minimalistic, or stylized deck option.
    • Card deck images are located in...: A static note providing the file path on the server where the selected card deck assets are stored for reference.

Interface

Purpose: Manages the general behavior and display elements of the user-facing platform, enhancing user experience and meeting legal requirements.

  • Key Parameters:
    • Main menu expanded: Only applicable for large screens, this toggle controls whether the primary site navigation menu is open by default or collapsed.
    • Display system bar: Toggles the visibility of a global system notification or status bar. (Note: Requires a broadcaster service to be set up, similar to the Chat feature).
    • Display users online status: Toggles the visual counter that shows the number of currently active users on the platform. (Note: Also requires a broadcaster service).
    • Require cookie consent: A critical legal toggle. If enabled, the site will display a banner or modal requiring the user to explicitly accept the use of cookies before proceeding, helping ensure compliance with data privacy regulations (like GDPR).
    • Enable sound by default: Controls the initial state of game sounds and music when a user first loads the platform or a game.

KYC (Know Your Customer)

Purpose: Configures the master control for the platform's identity verification process and anti-fraud checks.

  • Key Parameters:
    • Enabled: A master toggle switch that turns the entire KYC process on or off.
    • Provider: A dropdown menu to select the method used for processing and verifying submitted KYC documents:
      • Manual: Requires administrators to manually review and approve/reject all submitted documents (ID, proof of address, etc.).
      • BlueAssure: Integrates with an external, automated KYC provider (BlueAssure) for faster, third-party verification. Note: Setting this requires configuration of the BlueAssure API keys in the SERVICES section.
    • Restricted pages: A multi-select field used to enforce KYC compliance. It defines which parts of the site users are blocked from accessing until their verification status is complete (e.g., game, cashback, faucet, deposit, withdrawal).

Language

Purpose: Manages the localization and language selection behavior for the user-facing interface, ensuring content is displayed in the correct language based on user or browser settings.

  • Key Parameters:
    • Default language: A dropdown to set the base language the site will use if no user preference or browser setting is detected (e.g., English).
    • Detect default language from the browser: A toggle that, if enabled, prioritizes the language setting of the user's web browser when first loading the site, providing an automatic localization experience.
    • Allow language change: A toggle that controls whether users are permitted to manually change the display language of the front-end site using a selector tool.

Number format

Purpose: Defines how monetary and numerical values are displayed, ensuring consistency and localization across the entire platform's user interface and documentation.

  • Key Parameters:
    • Decimal separator: A dropdown to select the character used to separate the whole number from the fractional part (e.g., a period . for US/UK formats, or a comma , for European formats).
    • Thousands separator: A dropdown to select the character used to separate thousands in large numbers (e.g., a comma , for US/UK formats, or a period . or a space for European formats).

Session

Purpose: Manages user session longevity and security by defining how long a user's authenticated state remains valid.

  • Key Parameter:
    • Session lifetime: A field to set the duration, in minutes, for how long a user's login session remains active. After this time has passed without activity, the user will be automatically logged out for security purposes. The default value of 10080 minutes is equal to 7 days.

Users

Purpose: Controls core user profile settings and administrative auditing options.

  • Key Parameters:
    • Auto create an avatar for new users: If enabled, the system automatically generates and assigns a default visual icon or avatar to new accounts upon registration, ensuring every user has a distinct profile appearance from the start.
    • Allow to change the username: Toggles whether users are permitted to modify their display name after initial registration. Disabling this enforces username permanence for better security and moderation tracking.
    • Allow regular users to view other users profiles: Controls public profile visibility. If enabled, non-admin users can click on and view the basic profiles and activity of other players.
    • Enable admin audit log: A critical security feature. If enabled, the system creates a comprehensive log of every administrative action performed in the backend (e.g., manual debits, user edits, setting changes), providing an indispensable record for accountability and investigation.

User flags (1)

Purpose: Manages a specific set of customizable flags or tags for user accounts. These tags are primarily used for internal administrative categorization, such as identifying accounts that require special monitoring or attention.

  • Key Parameters:
    • ID: The internal, unique identifier for the flag (e.g., high_risk).
    • Title: The display name of the flag visible to administrators (e.g., High risk).
    • Color: The visual color code (e.g., #FF0000 for red) used to highlight the flag when applied to a user's account in the administrative panel.
    • Actions: Includes icons to Edit the flag details (ID, Title, Color) or Delete the flag entirely.
    • Add Flag: A button to create a new custom flag.

User sign-up form fields

Purpose: Configures which fields are mandatory or optional during the registration process, allowing administrators to capture custom data from new users.

  • Key Parameters:
    • Parameter name: A text input field where the administrator defines the internal name for the custom sign-up field.
    • + ADD Button: A button that is used to create and add the custom field to the sign-up form.
    • Description: A static note reminding the administrator that they can "Add any number of extra fields that should be present on the sign up form and filled in by the user."

Tips

Purpose: Configures the user tipping system, managing limits, frequency, and overall availability.

  • Key Parameters:
    • Enabled: A master toggle to activate or deactivate the entire user tipping feature globally.
    • Max amount: Sets the upper limit on the size of an individual tip a user can send (e.g., up to 500).
    • Interval: Sets the time frequency (e.g., every 24 hours) within which the Max amount limit is reset, controlling how often users can send high-value tips.

Vip club

Purpose: Manages the display and behavioral settings for the VIP and loyalty program features.

  • Key Parameter:
    • Enabled: A single toggle switch that acts as the master control to globally turn the entire VIP club and loyalty program features on or off.

CONTENT

The CONTENT section manages all regulatory and static non-dynamic text that is displayed prominently on the user-facing site.

  • Purpose: To manage legal documentation and static pages like "About Us."

Auth forms

The Auth forms subsection controls the background visual assets for the user registration and login pages.

  • Purpose: To customize the appearance of the authentication screens.
  • Key Parameter:
    • Background image: A file upload field to select the image that will be displayed in the background of the user login, registration, and password recovery forms (e.g., /images/auth/auth.webp).

External links

The External links subsection manages the creation and placement of custom links that direct users away from the platform.

  • Purpose: To configure links to external pages such as partner sites, regulatory bodies, or media coverage.
  • Key Parameters for each link:
    • Title: The display name for the link (e.g., External link).
    • Group: The menu or section where the link is meant to be displayed (e.g., About us).
    • Menu icon: A dropdown to select an icon (using the mdi-link standard) to visually represent the link in the menu.
    • URL: The target destination URL (e.g., https://example.com).
    • Management: Includes buttons to DELETE LINK and + ADD LINK.

Email templates

The Email templates subsection allows administrators to directly edit the HTML and content of the system's transactional and notification emails.

  • Purpose: To customize the branding, design, and messaging of automatic system emails.
  • Key Customization Areas (HTML Editor):
    • Welcome email: The full HTML content for the message sent to new users upon successful registration.
    • Message: A general-purpose email template used for various system notifications or password resets.
    • Management: Each template provides SAVE and DELETE buttons.

Footer

The Footer subsection controls the links, text, and structure of the static content displayed at the bottom of every page on the site.

  • Purpose: To inject the site's copyright, contact, and necessary legal links.
  • Key Parameter:
    • Content: A large text area for embedding the HTML structure of the footer (e.g., <primary-footer />), allowing for full customization of its appearance and links.

Home page

The Home page subsection is the key area for managing the content and dynamic visual elements of the primary landing page users see upon arrival.

  • Purpose: To configure the main marketing and display elements of the landing page, including welcome banners and the rotating image slider.
  • Key Configuration Areas:
    • Welcome card: Manages the display image for the user's personalized welcome section.
      • Image for authenticated users: The image shown when a user is logged in.
      • Image for guests: The image shown when the user is not logged in.
    • Content: A large text area for injecting the primary structural HTML elements of the homepage content section. This uses tags that reference dynamic content modules:
      • HTML content: The main editor field.
      • Placeholder content examples include: <welcome-card />, <bonus-cards />, and <original-game-list />.
    • Slider: Manages the configuration of the rotating image carousel, typically used for promotional banners.
      • Height on large screens / Height on small screens: Defines the pixel height of the slider for responsive viewing.
      • Transition: Selects the animation style between slides (e.g., Fade).
      • Navigation / Pagination / Dark overlay: Toggle switches for enabling control elements and visual styling on the slider.
      • Auto rotate slides: Toggles automatic advancement of the slides.
      • Interval: Sets the duration (in seconds) each slide remains visible before rotating.
      • Slide 1 / + ADD SLIDE: Controls the individual slides, allowing the administrator to add and configure the images, links, and content for each promotional banner in the carousel.

Menu

The Menu subsection is where administrators manage the structure and content of both the primary and static footer navigation elements.

  • Purpose: To control the flow and organization of the user interface by customizing site links.
  • Key Management Areas:
    • Main menu items: A collapsible field that manages the links displayed in the primary site navigation bar. The list can be reordered and includes standard items like:
      • Home
      • Leaderboard
      • Latest bets
      • High wins
      • High losses
      • Provably fair
      • Privacy policy
    • Footer menu items: A collapsible field that controls the links displayed in the static footer area, typically used for legal or informational pages. The current selected links are:
      • Provably fair
      • Privacy policy
      • Terms of use

Social

The Social subsection manages the creation and visibility of links to the platform's social media and contact channels.

  • Purpose: To configure and display external social media links on the user-facing site.
  • Key Configuration Areas:
    • Send us an email: An expandable field to configure the mailto: link for user contact.
    • Follow us on [Platform Name]: Expandable fields to input the specific URL or handle for platforms like Facebook, Instagram, Twitter, and Twitch.
    • + ADD SOCIAL NETWORK: A button to add and configure links for other social platforms not listed by default.

Static pages

The Static pages subsection allows the creation, management, and editing of all custom, standalone content pages on the site, often used for legal and informational documentation.

  • Purpose: To manage and host pages such as the "Provably Fair" description, "Privacy Policy," or "Terms of Use."
  • Key Configuration Areas:
    • Provably fair / Privacy policy / Terms of use: These are expandable sections for editing the content of pre-existing, critical static pages.
    • + ADD PAGE: A button to create a new, custom static page.
    • Parameters for an individual page (e.g., Provably fair):
      • Title: The display title of the page (e.g., Provably fair).
      • Group: A categorization field for organizing static pages (e.g., Policy).
      • Slug: The final part of the URL where the page will be accessed (e.g., https://www.google.com/search?q=https://birbgames.com/pages/provably-fair).
      • Menu icon: A dropdown to select an icon (e.g., mdi-certificate) to represent the link in any menus.
      • HTML content: A large text area for embedding the complete HTML structure and content of the page.

SERVICES

The SERVICES section is dedicated to configuring and connecting external application programming interfaces (APIs) and third-party tools.

  • Purpose: To integrate external tools for analytics, notifications, and other supporting features.
  • Key Configuration Areas:

Broadcaster

The Broadcaster settings configure the real-time push notification service used for features like chat, live bet streams, and online user status updates.

  • Broadcaster: A dropdown to select the type of broadcasting driver. Options are None, Pusher, or Reverb. Setting this up is crucial for enabling real-time features across the site.

Crypto API

The Crypto API manages the connection parameters for external services used to track cryptocurrency prices and facilitate conversions.

  • API: A dropdown to select the preferred crypto data provider (e.g., Cryptocompare).
  • API key: The field for entering the required key to authenticate with the chosen API service.
  • Management: Includes a REFRESH button to update data after changes and a GET A FREE KEY button to link to the provider's sign-up page.

Google Tag Manager

The Google Tag Manager configuration integrates the platform with Google's tag management system for analytics and marketing tracking.

  • Container ID: The field for entering the unique tracking ID (e.g., GTM-XXXXXX) used to manage marketing and analytics tags on the platform.

Google reCaptcha

The Google reCaptcha manages the keys required to enable the anti-bot security features on login and registration forms.

  • Version: A dropdown to select the reCaptcha version (e.g., 2 or 3).
  • Public key / Secret key: The two required API keys from Google for authenticating the reCaptcha service.

Post Affiliate Pro

The Post Affiliate Pro integration connects the platform's affiliate data to the external Post Affiliate Pro tracking system.

  • Script ID / Script URL: The specific identifiers and script source required for clicks tracking, usually found under "Tools -> Integration -> Clicks tracking" in the Post Affiliate Pro dashboard.
  • API URL: The URL endpoint for the Post Affiliate Pro API.
  • Sign up action code: The specific code used to record a new user registration event in the Post Affiliate Pro system.

Queues

The Queues configuration manages the underlying driver for the background processing system, which handles non-immediate tasks such as sending emails, processing large reports, or running maintenance scripts.

  • Purpose: Manages the configuration and driver for the background processing system.
  • Queue driver: A dropdown to select the underlying technology used to manage the job queue. Options include: Database, Redis, or Sync.
  • Note: The system provides a crucial reminder that the queue worker must be manually started as a separate background process for this feature to function properly.

Redis

The Redis section configures the connection details for the high-performance Redis in-memory data store.

  • Purpose: To configure the connection parameters for the external Redis service.
  • Key Parameters: URL, Host, Port, Database, and Password.

SMTP

The SMTP (Simple Mail Transfer Protocol) section manages the credentials and configuration necessary for the platform to send all outgoing system emails.

  • Purpose: To ensure reliable delivery of emails, including password resets, welcome messages, and administrative alerts.
  • Key Parameters: Host, Port, Encryption, User name, Password, E-mail from address, and E-mail from name.

Support chat

The Support chat section manages the integration of third-party live support chat widgets onto the user-facing site.

  • Purpose: To embed and configure external services that provide real-time customer support.
  • Key Parameters: Support chat provider (Options: None, Crisp, or Tawk).
